Innosim

Innosim is an InnoDB IO simulator that tests the disk IO capacity of a server. This started out as something I wrote on a Saturday to reproduce a performance problem for a vendor. It is meant to be a useful but simple approximation of InnoDB. As a simulator it chooses simplicity over accuracy. I highlight a few of those choices below. Some aspects of it (the shell scripts) definitely look like they were written in a hurry but it has turned out to be very useful. This document describes how it works using several perspectives: the files used during the test, the types of worker threads within the Innosim process, the steps used to run Innosim and the output produced by Innosim.

The file perspective

Innosim reads and writes multiple files to mimic the disk IO behavior of InnoDB and MySQL. These files are: database, binlog, trxlog.

·      Database – this represents the database file. Random page reads and writes are done to it. Sequential writes for the doublewrite buffer may also be done to it. The file is opened using O_DIRECT so fsync is not done after writes. Random reads and writes are done using pages of size –data-block-size bytes. Doublewrite buffer writes are done using large writes of 1 to 64 pages (up to 1MB for 16kb pages). The file to use is named by –data-file-name. The –doublewrite option can be used to disable doublewrite buffer writes. The –doublewrite-file-name option can be used to specify a different file to be used for the doublewrite buffer. When –doublewrite-file-name is not used then doublewrite writes use the first 64 pages of the database file.

·      Binlog – this is the log for MySQL replication. It is only written, the writes are –binlog-write-size bytes (default 128), and fsync is done after each write. The file uses buffered IO and fsync is done after each write. The file to use is named by –binlog-file-name. The file is truncated when –binlog-file-size bytes have been written to it.

·      Trxlog – trxlog is short for “transaction log”. This is the redo log for InnoDB. It is only written, the writes are –trxlog-write-size bytes (default 512), and fsync is done after each write. The file uses buffered IO and fsync is done after each write. The file to use is named by –trxlog-file-name. It is treated as a circular buffer. When –trxlog-file-size bytes have been written the file is then rewritten from the start but not truncated.

The process perspective

Innosim can also be described by the worker threads it uses to generate disk IO requests. There are three types of worker threads: user, scheduler, writer.

User threads model threads that handle user queries in MySQL. There are –num-users User threads. They may do disk reads from the database file and writes/fsyncs to the trxlog and binlog. They do not do writes to the database file. In MySQL/InnoDB user threads sometimes do writes but that complexity is not needed for Innosim. There is no think time and each thread runs a loop doing the following:

1.     If –read-hit-pct is not 0 then generate a random number (RN). If RN is greater than –read-hit-pct then read a randomly selected database page.

2.     Generate another random number (RN). If RN is less than –dirty-pct then increment the count of dirty pages, write/fsync the trxlog unless that has been disabled and write/fsync the binlog unless that has been disabled.

3.     Repeat

There is one Scheduler thread. It wakes up every –scheduler-sleep-usecs microseconds to request that dirty pages get written by the writer threads. If the dirty page count is non-zero when it is awake then it does the following until the count is zero and then it sleeps again. This is a simple approximation of InnoDB. It will not capture interesting behavior like checkpoint IO storms.

1.     Let P be the minimum of 64 and the count of dirty pages.

2.     Do a doublewrite buffer write for P times --data-block-size bytes.

3.     Generate P random page IDs and add them to the list of pages to be written. This list is consumed by the Writer threads.

4.     Repeat

Writer threads are similar to the background IO threads used by InnoDB. There are –num-writers Writer threads. The Writer threads wait for the scheduler threads to insert page IDs on the list of dirty pages. They remove items from that list and perform the writes.

The workflow perspective

The typical database file used for the test is large. I usually prepare that file once (--prepare 1) and then reuse it many times. If the prepare flag is not set to 1 then the database file must be at least as large as –data-file-size or the test will fail. Other files used by the test do not have this restriction.

After files have been prepared I use the run_all.sh script at http://bazaar.launchpad.net/~mdcallag/mysql-patch/mytools/files/head:/bench/innosim. This runs several variations of the test. All variations are run for –num-users 1, 4, 8, 16, 32 and 64. First the test is run in read-only mode (--dirty-pct 0 and –read-hit-pct 0) to determine the maximum IOPs that can be achieved. Then the test is run in write-only mode (--dirty-pct 100 and –read-hit-pct 100) to saturate any caches that exist in the storage system and generate garbage for storage cards that must do garbage collection (flash). Finally the test is run in read-write mode for several read:write ratios (8:1, 3:1, 2:1). I am usually most interested in the results for the read-write tests.

Some math must be done to get from the value used for –dirty-pct to the read:write ratio. In this case the read-write mode tests are done with –read-hit-pct 100 so when –dirty-pct X is used then there are X page writes for every 100 page reads. But if the doublewrite buffer is also enabled then each page is written twice – once using random in-place writes to the database file and once as part of a large sequential write to the doublewrite buffer. The doublewrite buffer is enabled for these tests so there are 2X page writes for every 100 page reads. As the read-write tests are run for –dirty-pct 6, 17 and 25 then there are 12, 34 and 50 writes per page read and the read:write ratios are approximately 8:1, 3:1 and 2:1. These ratios mean more for tests that use Flash cards than for tests that use spinning disk because for disk the random page writes cost much more than the doublewrite buffer writes.

The script run_all.sh invokes run_innosim.sh for each test configuration and run_innosim.sh invokes the innosim binary. When run_all.sh is used, the innosim binary is started 30 times or 5 X 6 times where there are 5 test modes (read-only, write-only, read-write 8:1, read-write 3:1, read-write 2:1) and 6 tests per mode (--num-users 1, 4, 8, 16, 32, 64).

After run_all.sh finishes, and that takes 15 hours when run using the arguments “0 1800 0” where 1800 means 1800 seconds per test, then I run stats.sh to generate reports. It takes 15 hours because innosim is invoked 30 times and each time takes 30 minutes (or 1800 seconds).

The reports generated by stats.sh are described in the next section.

The output perspective

Each time that run_innosim.sh is run it creates 4 files. All files have a common suffix that encode many of the test configuration variables. The output file is o.$suffix and this has STDOUT from innosim. It is the most useful file. The error file is e.$suffix and can usually be ignored. Output from vmstat during the test is in v.$suffix. Output from iostat during the test is in i.$suffix.

The stats.sh file assumes that run_all.sh was run. If you edit run_all.sh and change the concurrency levels used by the test then you must update stats.sh. These shell scripts are ugly. After running stats.sh there are 2 interesting files per test mode: report.X and percentile_iops.X where X is one of 0, 6, 17, 25 and 100. When X is 100 percentile_wiops.100 should be read instead of percentile_iops.100. X is the value of –dirty-pct used for the test. Thus, report.0 has results for the read-only test, report.100 has results for the write-only test and report.6, report.17 and report.25 have results for the read-write tests.

Help

 

Command-line options for innosim:

  --help -- print help message and exit

  --prepare 1|0         -- 1: create files and then run, 0: assume files exist, run

  --doublewrite 1|0     -- 1: use doublewrite buffer, 0: don't use it

  --binlog-file-name s  -- pathname for binlog file

  --trxlog-file-name s  -- pathname for transaction log file

  --data-file-name s    -- pathname for database file

  --doublewrite-file-name s -- pathname for doublewrite file, when not set use first 2MB of data file

  --binlog 1|0          -- 1: write to binlog, 0: don't write to it

  --trxlog 1|0          -- 1: write to transaction log, 0: don't write to it

  --binlog-write-size n -- size of binlog writes in bytes

  --trxlog-write-size n -- size of transaction log writes in bytes

  --binlog-file-size n  -- size of binlog file in bytes

  --trxlog-file-size n  -- size of transaction log file in bytes

  --data-file-size n    -- size of database file in bytes

  --data-block-size n   -- size of database blocks in bytes

  --num-writers n       -- number of writer threads

  --num-users n         -- number of user threads

  --max-dirty-pages n   -- maximum number of dirty pages, used to rate limit write-mostly workloads

  --dirty-pct n         -- percent of user transactions that dirty a page

  --read-hit-pct n      -- for transactions that dirty a page, percentage of time a read is not needed because it was cached

  --stats-interval n    -- interval in seconds at which stats are reported

  --test-duration n     -- number of seconds to run the test

  --scheduler-sleep-usecs n -- page writes are scheduled every n microseconds

Work in progress

 

More features are planned for innosim including:

·      Group commit for binlog and trxlog writes to amortize the fsync overhead

·      Support a non-uniform random number generator to determine blocks that are read and written

·      Split the database across a configurable number of files
